## 実装スコープの制限

このシステムは **Slackと連携するカンバン型タスク管理** に特化する。

### 実装しない機能
- ガントチャート
- カレンダービュー
- 工数管理・時間追跡
- スプリント管理（アジャイル）
- レポート・分析ダッシュボード
- メール通知（Slack通知のみ）

### 実装する機能（現在・計画中）
- カンバンボード（ドラッグ&ドロップ）
- Slack連携（タスク投稿・チャンネル選択）
- チームメンバーへのタスク割り当て
- コメント・ラベル機能
- セッションベース認証

## 2. OSS 運営方針

### Issue 管理

#### バグ報告テンプレート（`.github/ISSUE_TEMPLATE/bug_report.md`）
- 再現手順（必須）
- 期待動作 / 実際の動作
- 環境情報（OS, Node.js バージョン, ブラウザ）
- スクリーンショット（任意）

#### 機能要求フロー
1. GitHub Issue を `enhancement` ラベルで作成
2. ディスカッションで要件を整理（必要に応じて `discussion` ラベルを付与）
3. メンテナーが優先度を評価し、ロードマップに反映
4. `help wanted` / `good first issue` ラベルでコントリビューターを募集

#### ラベル一覧

| ラベル | 用途 |
|--------|------|
| `bug` | 不具合報告 |
| `enhancement` | 機能要求 |
| `good first issue` | 初心者向けタスク |
| `help wanted` | コントリビューター募集 |
| `documentation` | ドキュメント改善 |
| `security` | セキュリティ関連 |
| `wontfix` | 対応しないと決定 |

---

### PR レビューフロー

1. **作成**: `feature/xxx` ブランチから `main` へ PR を作成
2. **初回レビュー**: **1〜2 営業日以内**にメンテナーがレビュー開始
3. **修正対応**: レビューコメントへの対応後、再レビューを依頼
4. **マージ条件**:
   - メンテナー 1 名以上の承認
   - CI（lint / build）が全パス
   - コンフリクトなし
5. **マージ方式**: Squash merge を基本とする（コミット履歴を整理）

---

### ブランチ戦略

```
main
 └─ feature/xxx   （新機能開発）
 └─ fix/xxx       （バグ修正）
 └─ docs/xxx      （ドキュメント更新）
 └─ refactor/xxx  （リファクタリング）
```

| ブランチ | 役割 | マージ先 |
|----------|------|---------|
| `main` | 本番・リリースブランチ | — |
| `feature/xxx` | 機能開発 | `main` |
| `fix/xxx` | バグ修正 | `main` |
| `docs/xxx` | ドキュメント | `main` |

---

### リリース方針

**セマンティックバージョニング**（`MAJOR.MINOR.PATCH`）を採用する。

| 種別 | バージョン更新 | 例 |
|------|---------------|-----|
| 後方互換性のない変更 | MAJOR | 1.0.0 → 2.0.0 |
| 後方互換ある新機能追加 | MINOR | 0.1.0 → 0.2.0 |
| バグ修正 | PATCH | 0.1.0 → 0.1.1 |

**リリース手順**:
1. `CHANGELOG.md` を更新
2. バージョンタグ（`v0.2.0`）を作成
3. GitHub Releases にリリースノートを公開
4. `npm publish`（将来的にパッケージ化する場合）

## 4. 技術的負債と改善計画

### settei.json → DB 管理への移行

**現状の問題**:
- ユーザー情報が `settei.json` にハードコードされている
- メンバー追加・削除がファイル編集に依存しており、スケールしない

**移行計画**:

1. `users` テーブルを新設（`id`, `email`, `password_hash`, `role`, `created_at`）
2. 現行の `settei.json` ユーザーをマイグレーションスクリプトで DB に移行
3. 認証処理を DB 参照に切り替え
4. 管理 UI（ユーザー招待・削除画面）を実装（v0.2 と連携）

---

### テストカバレッジ向上

**現状**: テストが不十分

**目標**:

| フェーズ | 目標カバレッジ | 対象 |
|---------|--------------|------|
| v0.2 | 40% | API ルート・ユーティリティ関数 |
| v0.3 | 60% | コンポーネント・DB操作 |

**使用ツール**:
- ユニットテスト: Jest + React Testing Library

### TypeScript strict mode の強化

**現状**: `strict: false` または一部のルールのみ有効

**対応方針**:

1. `tsconfig.json` で `"strict": true` を有効化
2. 型エラーを段階的に修正（PR 分割で対応）
3. `any` の使用を原則禁止し、`unknown` + 型ガードに置き換え
4. 完了目標: v0.3 リリースまでに全ファイルを strict 対応

## 5. セキュリティ方針

### 脆弱性報告窓口

脆弱性を発見した場合は、**GitHub の公開 Issue には投稿せず**、以下に直接連絡してください：

- **メール**: taro@repeat.co.jp
- **件名フォーマット**: `[SECURITY] kanban-flow-oss - <概要>`
- **初回返信目標**: 3 営業日以内

報告内容には以下を含めてください：
- 脆弱性の概要と影響範囲
- 再現手順
- 推奨する修正案（任意）

修正完了後、報告者の同意のもとで `CHANGELOG.md` および GitHub Security Advisory に開示します。

---

### 定期的な依存関係アップデート

| 頻度 | 作業内容 |
|------|---------|
| 毎週 | `npm audit` を実行し、高リスクの脆弱性を即時対応 |
| 毎月 | `npm outdated` を確認し、マイナー・パッチアップデートを適用 |
| 四半期 | メジャーバージョンアップの影響評価・適用 |

**自動化**: GitHub Dependabot を有効化し、PR を自動作成する。

---

### セキュリティチェックリスト

`AGENTS.md` に記載済みのチェックリストを参照してください。主な確認項目：

- [ ] 環境変数・シークレットがコードにハードコードされていないか
- [ ] `settei.json` に実認証情報が含まれていないか（テンプレートのみ）
- [ ] SQL インジェクション対策（パラメータ化クエリの使用）
- [ ] XSS 対策（ユーザー入力のエスケープ）
- [ ] CSRF 対策（セッションクッキーの `SameSite` 設定）
- [ ] 依存パッケージに既知の脆弱性がないか（`npm audit`）
- [ ] HTTP ヘッダーのセキュリティ設定（`next.config.js` の `headers`）
